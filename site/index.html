Stack

Six layers, top to bottom.

Distribution
install.sh + GitHub Releases + Pages
A shell script resolves the latest release tarball from GitHub, drops it onto the machine, and installs a launcher at ~/.local/bin/guy. The public site is generated from structured content plus CHANGELOG.md and deployed by GitHub Pages.
CLI surface
guy install | status | doctor | repair | auth | sandbox
One binary routes every operator flow. No separate daemons, no hidden config UIs. If you can type guy, you can see what the runtime thinks is true and repair drift when it breaks.
Runtime kernel
@the-guy/core, @the-guy/doctor, @the-guy/sandbox
Core owns install state, asset rendering, stale asset pruning, profile resolution, and pack post-install tasks. Doctor turns hidden drift into explicit check results. Sandbox owns the Docker driver contract and warm-container lifecycle.
Payload boundary
@the-guy/pi-pack + generated profile mirror
The Pi pack is the canonical product payload. ~/.pi/** is disposable render output. profiles/power-user/assets is still generated as a founder-compatibility mirror, not the authoring source.
Source of truth
dotfiles-agents + pi-curated-skills → guy-pi-pack
First-party harness files come from dotfiles-agents/home/.pi/agent. Third-party curated skills come from pi-curated-skills exports. The Guy consumes both through pnpm sync:power-user-payload and records vendor provenance in upstream-skills.manifest.json.
Execution envelopes
native home · docker sandbox · future remote
guy install writes into a real macOS home. guy sandbox start builds the same payload into a Docker container with its own persistent home volume. Remote execution is a future driver over the same start/status/exec/doctor contract, not a shipped command yet.
External deps
pi · claude · codex · gh · docker
Pi is the managed requirement. Claude, Codex, GitHub CLI, and Docker are checked at doctor time but not force-installed. The product tells you what it needs instead of hiding the dependency.

Three paths through the stack, step by step.

Native install
Resolve bundle
install.sh downloads or reuses the release tarball and installs a launcher.
Run guy install
The TypeScript runtime resolves the power-user profile, loads the shipped Pi pack, and merges pack/profile post-install tasks.
Render runtime
Managed assets land in ~/.pi/**; state, logs, backups, and render metadata land in ~/.guy/**.
Prune stale assets
Files that used to be managed but disappeared from the next payload are removed during install.
Verify health
Doctor checks confirm runtime state, managed assets, Pi readiness, platform support, and provider auth handoffs.
Docker sandbox
Start sandbox
guy sandbox start rebuilds a bundle-style image context from the current repo or installed release.
Build image
Docker bakes the runtime bundle plus Pi into an image tagged for the current version.
Warm container
A named container and persistent home volume come up with the repo mounted as /workspace.
Bootstrap inside
The container runs guy install or guy repair in its own home so the sandbox stays self-consistent.
Operate
Humans use shell; automation and future control planes use exec, status, and doctor.
Payload sync
Update source
First-party changes land in dotfiles-agents; curated third-party skills land in pi-curated-skills.
Render curated skills
pi-curated-skills produces exported skill payloads plus provenance metadata.
Sync into The Guy
pnpm sync:power-user-payload copies allowed source paths into packages/guy-pi-pack and mirrors them into profiles/power-user/assets.
Verify
pnpm test checks the pack manifest, post-install tasks, schema, runtime, CLI, and sandbox driver.
Release + site publish
Tag release
GitHub Actions runs tests, builds the bundle, publishes assets, and creates release notes.
Refresh changelog
The release workflow writes release notes back into CHANGELOG.md.
Rebuild this page
The site generator re-renders from structured content plus the latest changelog entries.
Deploy
GitHub Pages publishes the architecture story and release ledger without manual editing.

Bets

Why one payload?
Native and sandbox should not diverge. One pack, one doctor, one repair path.
Why Docker first?
Fastest to ship. Driver seam exists for SSH, OpenShell, or VMs later.
Why not sync ~/.pi directly?
Rendered files drift. The product payload must be reproducible from source repos plus explicit metadata.
Why vendor-skills?
Third-party skills need provenance and dependency setup without pretending The Guy authored them.
Why release notes here?
If the public page doesn't update from the changelog, the story drifts.
