Stack

Six layers, top to bottom.

Distribution
install.sh + GitHub Releases
A shell script resolves the latest release tarball from GitHub, drops it onto the machine, and installs a launcher at ~/.local/bin/guy. That is the entire job of bash. Everything else is TypeScript.
CLI surface
guy install | status | doctor | repair | auth | sandbox
One binary routes every operator flow. No separate daemons, no hidden config UIs. If you can type guy, you can see what the runtime thinks is true.
Runtime kernel
@the-guy/core, @the-guy/doctor, @the-guy/sandbox
Core owns install state, asset rendering, and profile resolution. Doctor turns hidden drift into explicit check results. Sandbox owns the Docker driver contract and warm-container lifecycle.
Payload
@the-guy/pi-pack + profiles/power-user
The Pi pack is the canonical home of shipped runtime files — agents, prompts, skills, extensions, settings. Profiles are presets that choose which pack to use and which tools to manage. ~/.pi/** is disposable render output, not a source of truth.
Execution envelopes
native home · docker sandbox · (future remote)
guy install writes into a real macOS home. guy sandbox start builds the same payload into a Docker container with its own persistent home volume. The future Paperclip remote surface will use the same exec contract without inventing a new packaging model.
External deps
pi · claude · codex · gh · docker
Pi is the managed requirement. Claude, Codex, GitHub CLI, and Docker are checked at doctor time but not force-installed. The product tells you what it needs instead of hiding the dependency.

Three paths through the stack, step by step.

Native install
Resolve bundle
install.sh downloads or reuses the release tarball and installs a launcher.
Run guy install
The TypeScript runtime resolves the power-user profile and loads the shipped Pi pack.
Render runtime
Managed assets land in ~/.pi/**; state, logs, backups, and render metadata land in ~/.guy/**.
Verify health
Doctor checks confirm runtime state, managed assets, Pi readiness, and provider auth handoffs.
Docker sandbox
Start sandbox
guy sandbox start rebuilds a bundle-style image context from the current repo or installed release.
Build image
Docker bakes the runtime bundle plus Pi into an image tagged for the current version.
Warm container
A named container and persistent home volume come up with the repo mounted as /workspace.
Bootstrap inside
The container runs guy install or guy repair in its own home so the sandbox stays self-consistent.
Operate
Humans use shell; automation and future control planes use exec, status, and doctor.
Release + site publish
Tag release
GitHub Actions runs tests, builds the bundle, publishes assets, and creates release notes.
Refresh changelog
The release workflow writes release notes back into CHANGELOG.md.
Rebuild this page
The site generator re-renders from structured content plus the latest changelog entries.
Deploy
GitHub Pages publishes the architecture story and release ledger without manual editing.

Bets

Why one payload?
Native and sandbox should not diverge. One pack, one doctor, one repair path.
Why Docker first?
Fastest to ship. Driver seam exists for OpenShell or VMs later.
Why release notes here?
If the public page doesn't update from the changelog, the story drifts.
