# Payload sync workflow

The Guy ships Minzi's Pi harness as a product payload. The shipped files in this repo are **consumer copies**, not the authoring source of truth.

## Source of truth

| Kind | Authoring source | The Guy destination | Notes |
|------|------------------|---------------------|-------|
| First-party Pi harness files | `../dotfiles-agents/home/.pi/agent` or `$THE_GUY_PI_SOURCE` | `packages/guy-pi-pack/assets/.pi/agent/**` | `AGENTS.md`, `settings.json`, agents, prompts, selected extensions, first-party skills. |
| Curated third-party skills | `../pi-curated-skills/exports` or `$THE_GUY_PI_CURATED_SKILLS_SOURCE` | `packages/guy-pi-pack/assets/.pi/agent/vendor-skills/**` | Generated exports with provenance and post-install metadata. |
| Product payload manifest | generated by `pnpm sync:power-user-payload` | `packages/guy-pi-pack/pack.json` | Canonical pack manifest for the runtime. |
| Founder compatibility mirror | generated by `pnpm sync:power-user-payload` | `profiles/power-user/assets/**` and `profiles/power-user/assets.json` | Temporary fallback mirror. Do not edit by hand. |

## What syncs from `dotfiles-agents`

`scripts/sync-power-user-pi-payload.mjs` copies this subset from the first-party source root:

- files: `AGENTS.md`, `settings.json`
- directories: `agents/`, `prompts/`, `skills/`
- selected extensions:
  - `extensions/_shared/`
  - `extensions/context-awareness.ts`
  - `extensions/session-browser/`
  - `extensions/subagent/`

The sync intentionally excludes:

- `skills/linear` for now
- `node_modules`, `.git`, `dist`

Repo-owned exceptions are restored after copy:

- `skills/team-debate/SKILL.md`

## What syncs from `pi-curated-skills`

The curated repo owns third-party skill exports. The Guy reads:

```text
../pi-curated-skills/exports/pi-agent-skills.manifest.json
```

Each exported skill is copied under:

```text
packages/guy-pi-pack/assets/.pi/agent/vendor-skills/<skill-id>
```

The manifest also carries:

- source repo/path/ref
- pinned commit
- license
- replaced legacy skill ids
- post-install tasks

Example: `browser-tools` ships as a vendor skill, replaces `agent-browser`, and declares `npm ci` as a pack post-install task.

## Update loop

When harness changes land in `dotfiles-agents`:

```bash
cd /Users/minzi/Developer/the-guy
pnpm sync:power-user-payload
pnpm test
```

When third-party curated skills change:

```bash
cd /Users/minzi/Developer/pi-curated-skills
npm run render
npm run verify

cd /Users/minzi/Developer/the-guy
pnpm sync:power-user-payload
pnpm test
```

If the source repos are not siblings of The Guy, set explicit roots:

```bash
THE_GUY_PI_SOURCE=/path/to/dotfiles-agents/home/.pi/agent \
THE_GUY_PI_CURATED_SKILLS_SOURCE=/path/to/pi-curated-skills \
pnpm sync:power-user-payload
```

## Commit rule

Treat sync output as generated product payload:

1. Commit source changes in the source repo first (`dotfiles-agents` or `pi-curated-skills`).
2. Run The Guy sync.
3. Review generated diffs in:
   - `packages/guy-pi-pack/assets/.pi/agent/**`
   - `packages/guy-pi-pack/pack.json`
   - `profiles/power-user/assets/**`
   - `profiles/power-user/assets.json`
4. Commit The Guy payload refresh separately from unrelated runtime/site changes.
5. Open a PR and let Claude/Codex reviewers check the generated payload boundary.

## Why not sync automatically from CI?

The Guy should not silently publish whatever happens to be on a local harness branch. Payload promotion is a product decision, not a background mirror.

CI should verify generated state eventually, but promotion should stay explicit:

```bash
source repo change -> render/verify -> The Guy sync -> test -> PR -> release
```
